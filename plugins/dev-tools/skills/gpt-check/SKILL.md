# GPT Cross-Check (Codex CLI連携)

Claudeの成果物を独立した外部モデル(GPT / Codex CLI)にレビューさせ、見落とし・抜け漏れ・論理欠陥の指摘を受けて反映するワークフロー。

## 前提

- `codex` CLIがインストール済みで、ChatGPTアカウントでログイン済みであること。
- 利用はユーザーのChatGPT Plus枠(5時間ウィンドウ+週次上限)を消費する。呼ばれたときだけ実行すること。
- 実行環境によって `codex` コマンドの見つけ方・ログイン方法が異なる(下記「環境別の注意」参照)。まず `codex --version` で使えるか確認し、使えなければ環境別の対処を行う。

### 環境別の注意

**ローカル(Windows / Claude Code CLI)の場合:**
- インストール先は通常 `%APPDATA%\npm\codex`。Bashツールから呼ぶ場合はPATHに以下をPOSIX形式で追加する(Windows形式のパスを混ぜると `MODULE_NOT_FOUND` になるため厳禁):
  ```bash
  export PATH="$PATH:/c/Program Files/nodejs:/c/Users/hoyah/AppData/Roaming/npm"
  ```
- 認証エラーが出たら「別ターミナルで `codex login` を実行してください」とユーザーに案内する。

**claude.ai/code (Web版) など隔離クラウドサンドボックスの場合:**
- `codex` が未インストールなら、環境のセットアップスクリプトで `npm install -g @openai/codex` を実行するようユーザーに案内する(スナップショットされ次回以降は再インストール不要)。
- ネットワーク許可リスト(Custom domains)に `openai.com` 系ドメインを追加する必要がある。デフォルトのTrustedリストには含まれない。
- ブラウザのローカルコールバック(`codex login`)が使えないヘッドレス環境のため、`codex login --device-auth` を使う。認証トークンがセッションをまたいで永続化されるかは環境依存で未検証。失敗したら手順を中断し、ユーザーに状況を報告する。

## 手順

### 1. レビュー対象を特定する

- ユーザーが対象を指定していればそれ(ファイル、直前の成果物、diffなど)。
- 指定がなければ、この会話で直近に生成した成果物。
- 対象をスクラッチパッド(または一時ディレクトリ)に `draft.md`(コードなら適切な拡張子)としてUTF-8で書き出す。会話上にしか存在しない成果物は必ずファイル化する。

### 2. レビュープロンプトを作成する

`review-prompt.md` を書く。構成:

```
あなたは独立した厳格なレビュアーです。以下の成果物は別のAIが生成したものです。
作成者に忖度せず、批判的に監査してください。

## 監査観点
1. 抜け漏れ: 本来含まれるべきなのに欠けている論点・選択肢・リスク・エッジケース
2. 論理欠陥: 根拠と結論の飛躍、矛盾、暗黙の前提
3. 事実の疑義: 誤りの可能性がある記述(疑わしい箇所の指摘でよい)
4. 改善提案: より良い代替案があれば

## 出力形式
指摘ごとに: [重大度: 高/中/低] 指摘内容 / 該当箇所 / 修正の方向性
重大度順に列挙。問題がない観点は「問題なし」と明記。
指摘がゼロの場合も、最も弱い部分を1つ挙げること。

## 成果物
(ここに draft.md の内容を貼り込む。または「<パス> を読んでレビューせよ」と書く)
```

対象がコードやプロジェクト一式の場合は、内容を貼り込む代わりに対象ディレクトリを `--cd` で指定し、プロンプトには「◯◯を読んでレビューせよ」と書く方がよい。

### 3. Codexを実行する

Bashツール(またはシェル)で実行する:

```bash
codex exec - --sandbox read-only --skip-git-repo-check < "<review-prompt.mdのパス>"
```

`codex` がPATHに無ければ、上記「環境別の注意」に従ってPATHを補うか、セットアップを案内する。

- `--sandbox read-only`: Codexにファイル変更をさせない(レビュー専用)。必須。
- `--skip-git-repo-check`: gitリポジトリ外でも動かすため。必須。
- コード一式のレビュー時は `--cd "<対象ディレクトリ>"` を追加。
- タイムアウトは長めに(300000ms以上)。数分かかることがある。
- 認証エラー(login required等)が出たら手順を中断し、環境に応じたログイン方法をユーザーに案内する。

### 4. 指摘をトリアージして反映する

GPTの指摘を鵜呑みにしない。各指摘について:

1. 自分で妥当性を検証する(事実確認・コードなら該当箇所を読む)。
2. 妥当な指摘 → 成果物に反映する。
3. 誤検出・的外れ → 反映せず、理由を添えて記録する。

### 5. ユーザーに報告する

最終メッセージに必ず含める:

- **GPTの指摘一覧**(重大度つき、原文の要旨)
- **各指摘への対応**: 反映した/しなかった+理由
- **修正後の成果物**(または修正済みファイルへのリンク)

指摘を隠したり、都合の悪い指摘を省略したりしない。GPTとClaudeで見解が割れた場合は両論を提示してユーザーに判断を委ねる。

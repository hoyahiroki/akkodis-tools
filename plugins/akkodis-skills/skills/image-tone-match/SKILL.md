# image-tone-match

> **■ AKKODiS 本番設定（このパッケージの既定）**
> - 合わせ先＝合成ニュートラル基準（`assets/neutral_profile.json`）＝**中立トーン**で確定。
> - `--wb-strength` の既定を **0.7** に設定済み（背景の強い色被りで顔が青白くなる過補正を防ぐ）。引数なしのbareコマンドで本番挙動になる。
> - 色被りが極端な外れ値（金色の壁・夕日直射など）**だけ**個別に `--wb-strength 0.5〜0.6` を付けて再変換。
> - **運用フロー：事例・対談記事のMV写真は「撮影ガイドライン準拠で撮影 → 納品 → 本スキルで変換 → 公開」。指示書通りに撮ったカットも含め、全カットを毎回必ず通す**（撮影で揃うのは“ほぼ”であり、最後の1点合わせは本スキルが担う）。
>
> 標準コマンド（中立＋wb0.7で一括変換）: `python3 scripts/color_match.py <納品フォルダ> --out-dir <出力フォルダ>`
> 検収のみ（変換せずPASS/WARN/FAIL判定）: `python3 scripts/color_match.py <納品フォルダ> --check`

入力画像の **明度（L）/ 彩度（chroma）/ 色調・ホワイトバランス（a,b）** を1枚の**基準画像**に
合わせ、撮影条件のバラバラな写真群を「ひとつの見た目」に統一する。Web記事・ポータル・
レポート内の写真をそろえる用途を想定。

**動作はモード1つに固定：各画像を合成ニュートラル基準（既定）へ1対1で合わせる。** 入力Aの結果は、同じバッチに何を
入れても変わらない（バッチ非依存＝**決定論的・再現可能**）。複数人で使っても齟齬が出ないことを
優先した設計。色（WB）・黒点・白点・彩度は基準にマッチする。**ただし明るさ(中間調)だけは基準の
値を使わず、固定の目標へ寄せる**（理由は「明るさの扱い」節）。

手法は **CIELAB 統計転送**：各画像を CIELAB に変換し、チャンネルごとに平均・標準偏差が
基準と一致するよう再スケールする。`L` が明度/コントラスト、`a`/`b` の平均がホワイトバランス/
色被り、`a`/`b` の広がり（chroma）が彩度を担う。変えるのは色統計だけで、空間情報（顔・輪郭・
レイアウト）は触らない。a/b・彩度の倍率は **0.75〜1.35 に制限**（白壁中心のシーンで肌の赤みが過増幅
されたり、派手な背景で血色が抜けたりする「顔色の事故」を防ぐ顔色ガード）。**画素を作り直さない＝顔・ロゴ・文字は本物のまま**（生成AIのように描き直さない）。

## 同梱の基準＝合成ニュートラル・プロファイル（既定）

このスキルの既定の合わせ先は**写真ではなく合成ニュートラル・プロファイル**（`assets/neutral_profile.json`）。
特定シーンのクセを持たない「理想の中立点」で、各画像をこれに1対1で合わせる。

- WB：**ニュートラル**（a=0.5, b=2.0 ＝肌が自然に見える程度のごく僅かな暖色）
- 彩度：**ナチュラル**（chroma≈14。くすみすぎ・派手すぎない）
- 黒点3 / 白点96 ＝**フルレンジ**（締まった黒・抜けた白）
- 明るさは別途、固定目標(L中央値62)へ lift-only（暗いカットだけ持ち上げ）

ニュートラルなので、入力がどんな色被り（暖色/寒色/タングステン等）でも結果は素直に中立へ寄る。
**画素は作り直さない＝顔・ロゴ・文字は本物のまま**。固定JSONゆえ誰がやっても同一＝決定論的。

このスキルは**合成ニュートラル基準のみ**を使う（特定画像を基準にする機能は持たない）。狙ったハウス・
トーン（中立から少し暖色など）にしたい場合は `assets/neutral_profile.json` の数値を編集する（全体に反映）。

## ワークフロー

1. 合わせ先は常に合成ニュートラル基準（同梱の固定プロファイル）。指定不要。
2. 対象画像を1フォルダにまとめる（またはファイルを直接渡す）。
3. マッチを実行。必要なら `--print-stats` と比較画像（モンタージュ）で確認する。
4. 出力フォルダを提示する。

## コマンド

スクリプトは `scripts/color_match.py`。入力はファイル・フォルダ混在可。スキルのディレクトリ内で
実行する（またはスクリプトを絶対パスで指定）。

補助ツール `scripts/check_tone.py` は補正ロジックには一切触れない独立チェッカーで、素材の
L(明度)中央値とハイライトクリッピング率(%)を一覧表示する（`color_match.py --check` と同じ指標）。
撮影・書き出し後の素材が明るすぎ／暗すぎ・白飛びしていないかを素早く確認したい時に使う。
```bash
python3 scripts/check_tone.py /path/to/images            # 表で確認
python3 scripts/check_tone.py /path/to/images --csv       # CSV出力
```

**標準 — フォルダを同梱基準に合わせる（推奨デフォルト）：**
```bash
python3 scripts/color_match.py /path/to/images --out-dir /mnt/user-data/outputs/toned --print-stats
```

**彩度だけ控えめに寄せる（緑/紅葉など色の濃い写真でくすみ過ぎる時に有効）：**
```bash
python3 scripts/color_match.py /path/to/images --sat-strength 0.6 --out-dir /mnt/user-data/outputs/toned
```
`--sat-strength` は明度・コントラスト・WB（黒点アンカーと色被り補正）には一切触れず、
彩度の寄せ具合だけを 0..1 で調整する（1.0=基準の彩度に合わせる＝既定、小さいほど元の鮮やかさを残す）。

**明度の合わせ方を切り替える：**
```bash
# 既定: 黒点/白点を基準に合わせる（暗部の浮き=ヘイズを防ぐ。下記参照）
python3 scripts/color_match.py /path/to/images --l-mode anchor --out-dir /mnt/user-data/outputs/toned
# 従来: 平均+標準偏差マッチ（黒が浮きやすいので非推奨。意図がある時だけ）
python3 scripts/color_match.py /path/to/images --l-mode moments --out-dir /mnt/user-data/outputs/toned
```

**強度を調整（基準へ60%だけ寄せる＝各写真の個性を残す。全軸に効く）：**
```bash
python3 scripts/color_match.py /path/to/images --strength 0.6 --out-dir /mnt/user-data/outputs/toned
```

**出力形式を上書き（既定は WebP に統一。png/jpg などを指定すればそれに従う）：**
```bash
python3 scripts/color_match.py /path/to/images --format png --quality 92 --out-dir /mnt/user-data/outputs/toned
```
（`--format keep` で入力ごとの元形式を維持することも可能。）

出力は `--out-dir` に `STEM_toned.EXT` で書き出す。**入力形式は何でも受け付けるが、出力は既定で
WebP に統一する**（ユーザーが `--format png` / `--format jpg` 等を指定した場合はそれに従う）。
アルファチャンネルは保持（JPEG出力時を除く）。読み込み時に EXIF の回転を反映する。

## 手法の選び方（`--method`）

既定は `lab-moments`。ほぼ全ケースでこれが正解。理由のあるときだけ切り替える。

| 手法 | 合わせるもの | 使う場面 |
|---|---|---|
| `lab-moments`（既定） | L=黒点/白点アンカー＋ a,b の平均+標準偏差 | 写真セットの一般的な統一。堅牢・構造保存。 |
| `axes` | 軸別・個別トグル | 外科的制御。例：露出とWBはそろえるが各写真の彩度は**残す**。 |

`axes` のトグル：`--no-brightness` / `--no-white-balance` / `--no-saturation` /
`--saturation-std`（chroma の広がりも合わせる。既定は平均のみ）。`axes` のWBマッチは
**色相を回転させず**ニュートラル点だけを移動するので、肌色が破綻しにくい。

`--clip-ratio`（既定 3.0）は単一チャンネルのストレッチ上限。分散の極端に小さい素材での
破綻を防ぐ。

## 明度(L)の合わせ方（`--l-mode`）と彩度ダイヤル（`--sat-strength`）

**既定は `--l-mode anchor`（黒点/白点アンカー）。これが品質の要。** 入力 L の下位/上位
パーセンタイル（既定 p1/p99）を基準の同じ点へ線形写像し、「黒は黒・白は白」を保つ。

なぜ既定がこれか：平均+標準偏差マッチ（`moments`）だと、明るい背景が画面の多くを占める
基準（壁が大半・L平均が高い）に引きずられて、**暗部が L=10〜30 まで浮き、スーツや影が
乳白色にかすむ（ヘイズ）**。実写5枚のテストで全カットにこの症状が出た。アンカー方式は黒点を
基準の L≈3.7 に固定するのでヘイズが出ず、明るさを背景の白壁量に左右されないぶん内容差にも頑健。
従来挙動が必要なときだけ `--l-mode moments`。アンカー点は `--anchor-lo` / `--anchor-hi` で調整可。

**彩度は `--sat-strength`（0..1, 既定 1.0）で独立に制御。** 同梱基準は低彩度（chroma≈8）
なので、全補正だと緑/紅葉など色の濃い写真は彩度が大きく落ちる。`--sat-strength` は明度・
コントラスト・WB（黒点アンカーと色被り補正）に**触れずに**彩度の寄せ具合だけを下げるので、
「露出と色被りは完全統一、彩度は元の鮮やかさを一部残す」ができる。緑の多いオフィス/屋外
ロケのセットでは **0.5〜0.7 を推奨**（既定の 1.0 は基準のミュートな質感に忠実に合わせる側）。

## 明るさ(中間調)の扱い ＋ 仕上げツール

このスキルは**常に基準（既定＝合成ニュートラル）へ1対1で合わせる**（モードは1つに固定）。色（WB）・
黒点・白点・彩度は基準にマッチするが、**明るさ(中間調)だけは基準の値を使わない**。ハイキーな基準の
中央値に明るさを合わせると全体が白っぽく(washed)なるため、固定の目標値を使う。

そこで明るさは**固定の目標(既定L中央値62)へ lift-only で寄せる**：
- `--midtone-strength`（既定0.5）：寄せる強さ。**lift-only=目標より暗いカットだけ持ち上げ**、明るい
  良露出のカットは触らない。黒点・白点はガンマで固定するのでヘイズは出ない。0で明るさ非調整(色だけ合わせる)。
- `--midtone-target`（既定62, 固定値）：目標L中央値。**固定値なのでバッチに何を入れても結果は不変**＝
  決定論的。もっと明るく/暗くは 66 / 56 等。

**仕上げツール（非破壊・生成なし。質感だけを整える。すべて 0 で無効化可）：**
- `--fill`（既定0）：シャドウリフト。黒点を保ち暗部〜中間下部だけ持ち上げる。**逆光・明るい背景で
  被写体(黒スーツ等)が沈む写真に 0.3〜0.6**。グローバルな明るさ補正では救えない「背景は明るいが
  人物が暗い」構図に効く（ただし下記の天井あり）。
- `--highlights`（既定0）：ハイライト復元。明るい階調の最上部だけを引き下げ、**白飛び/眠さ(washed)
  を抑える**。明るすぎ・眠いと感じたら 0.3〜0.6。中間調・黒点は不動。
- `--clarity`（既定0.12）：ローカルコントラスト。立体感。肌のテカり/シワが出すぎたら 0.05〜0.08。
- `--contrast`（既定0.10）：S字トーンカーブ。端点固定でメリハリ。
- `--vibrance`（既定0）：肌色を保護した彩度持ち上げ。くすみが気になる時 0.1〜0.2。
- `--sharpen`（既定0）：Web表示の精細感。掛けるなら 0.2〜0.4。

**推奨レシピ（記事用の人物セット）：フラグ無しでOK（既定＝合成ニュートラル＋彩度0.85＋fill0.15）**
```bash
python3 scripts/color_match.py /path/to/images --out-dir /mnt/user-data/outputs/toned --print-stats
```
色・黒点をニュートラル基準へ揃え、暗いカットだけ自然に持ち上げ、クラリティ/コントラストで質感を足す。
**白すぎ/眠いと感じたら**：`--fill 0`、`--highlights 0.4`、または `--midtone-strength 0.4`。
**人物が暗い逆光カット**は `--fill 0.5`。

## 重要な限界 — ここは正直に伝える。ごまかさない

- **これは統計マッチであって、意味を理解したカラーグレーディングではない。** 入力が基準と
  内容的に近いことを前提にする。同梱基準は人物＋ニュートラル内装なので、人物/対談/チーム写真
  には強い。色構成が大きく異なる素材 — 白背景の製品・スクショ・UI・料理ドアップ・明るい屋外/
  風景 — を、くすんだベージュの人物プロファイルへ無理に寄せると**色が破綻する**。その種の素材
  には内容に合った基準を与えるか、`--strength` を下げるか、`axes`＋`--no-saturation` で回す。
- **彩度の落ちすぎは仕様であって不具合ではない。** 基準が低彩度のため。色の濃い写真がくすむと
  感じたら `--sat-strength 0.5〜0.7`。WB・黒点はそのまま維持される。
- **「全カットのL中央値を一致させる」ことは目標ではない。** 背景の明暗比は写真ごとに違う(明るい窓が
  大半の逆光カットは中央値が高くなる)。それを無理に同一中央値へ寄せると、明るい良露出の写真が暗く沈み
  かえって悪化する。だから lift-only＝暗いカットだけ持ち上げる方針を既定にしている。揃えるべきは
  WB・黒点・白点・コントラスト・彩度・被写体の適正露出であって、グローバル中央値の数値ではない。
- **被写体だけを明るくする「意味的な露出」はできない（原理的な天井）。** 「背景は明るいが人物が暗い」
  逆光カットは、グローバルなトーン操作では人物だけを起こせない（`--fill` で暗部全体は持ち上がるが
  限界がある）。ここは生成AIが意味を理解して描き直す＝**捏造**で解いている領域。非破壊で詰めるなら
  該当カットだけ Lightroom 等で部分補正(ドッジ/露出)するか、撮影時に被写体露出を合わせるのが正道。
- **WebP/JPEG は非可逆。** できるだけ高品質な原本から1回だけ書き出す。圧縮済み出力を繰り返し
  再トーンしない。品質重視なら中間は PNG が無難。
- **強い色被りの中立化は「顔」も道連れにする（構造的限界）。** 補正は画像全体の統計で行うため、
  金色の間接照明など環境光自体が強く色被りしたカットでは、背景を中立化すると顔の色も同方向へ
  動く（例: 暖色b=20の環境で顔が青白くなる）。緩和は `--wb-strength 0.4〜0.6`（色被り除去を部分適用、
  雰囲気と顔色を残す）。**根治は撮影**: 同梱 `assets/shooting_requirements.md` の要件で撮ってもらう。
- **意図的な色の雰囲気も中立化される。** 演出の暖色/寒色を残したいカットは `--sat-strength` を上げる
  か `--wb-strength` を下げる。逆に言えば、どんな色被りでも素直に中立へ揃うのが既定の利点。
- **明るさは基準に合わせない（固定目標へ lift-only）。** 基準のハイキー中央値(≈85)に明るさを
  合わせると washed になるため。固定値なのでバッチ非依存＝再現可能。

## 納品検収（--check）と撮影要件

撮影起因の問題（強い色被り・逆光の顔・白飛び）は後処理で完全には直せない。**撮影時に潰すのが正道**。
- 撮影要件: `assets/shooting_requirements.md`（カメラマンへ渡す指示書と同内容。WB/露出/階調/光源/背景/納品仕様）。
- 受入チェック: 変換せず品質判定だけ行う。
```bash
python3 scripts/color_match.py 納品フォルダ --check
```
WB色被り・黒の締まり・白飛び率・露出・彩度を `assets/shooting_spec.json` の閾値で
PASS/WARN/FAIL 判定し、不合格項目には撮影側の対処を表示する（指示書の検収基準と同一数値）。
ユーザーが「納品写真をチェックして」「検収して」と言ったらこのモードを使う。

## 結果の検証

`--print-stats` で各画像の CIELAB サマリを補正前後で表示する。補正後の値が基準プロファイルへ
収束しているはず。視覚確認には、PIL で「基準｜補正前｜補正後」の横並びを作り、全バッチ確定前に
ユーザーへ見せる。

# Output Critic — 成果物批評スキル

## 目的と設計思想

生成モデルは自分の出力を過大評価する（self-preference bias）。このスキルは、そのバイアスを構造的に排除するための**批評専用プロトコル**である。役割は検察官であり、弁護人ではない。ドラフトの良い点を褒めることに価値はない。価値があるのは、納品後にクライアントや役員から指摘されるはずだった欠陥を、納品前に発見することだけである。

このスキルが動作する原則：

1. **生成と批評の分離** — 批評者はドラフトの執筆意図を弁護しない。書かれていないことは存在しないものとして扱う。「おそらくこういう意図だろう」という補完は評価の敵である。
2. **修正版は書かない** — 出力は指摘と修正方針まで。リライトは生成側チャットの仕事。批評者がリライトを始めると、批評者自身のバイアスが最終案に混入し、分離の意味が消える。
3. **指摘の水増し禁止** — 重大度を偽装した些末な指摘でリストを膨らませない。指摘ゼロなら、ゼロと判定した根拠を明記して合格を出す。「何か言わなければ」という圧力に屈した指摘はノイズであり、差し戻しコストを増やすだけである。

## 入力要件

批評には以下の2点が必要：

| 入力 | 必須度 | 欠落時の挙動 |
|---|---|---|
| ドラフト本体（貼付・添付いずれも可） | 必須 | 批評不能。提供を求める |
| 元の要件（依頼内容・目的・想定読者・制約） | 強く推奨 | 要件不明のまま評価する旨を判定サマリ冒頭に明記し、網羅性評価（A系）の確度が下がることを宣言する |

添付ファイルがある場合は全文を読み切ってから批評を開始する。部分読みでの批評は誤検出の温床になる。読み切れない場合はその旨を先に明示する。

## 批評プロトコル（2パス方式）

### パス1：検出（網を広げる）

ドラフト全体を以下のルーブリックに対して走査し、候補となる指摘をすべて挙げる。この段階では偽陽性を恐れない。

**A. 網羅性（Coverage）**

| ID | 観点 | 検出する欠陥 |
|---|---|---|
| A1 | 論点漏れ | 要件・目的に照らして扱われるべき論点が欠落している |
| A2 | 選択肢の抜け | 提示された選択肢の集合に、明白に検討すべき代替案が含まれていない |
| A3 | 反例・例外の欠落 | 主張が成立しないケース・エッジケースへの言及がない |
| A4 | 前提条件の未記載 | 結論が特定の前提に依存しているのに、その前提が明示されていない |
| A5 | ステークホルダー視点漏れ | 想定読者・意思決定者・実行者のうち、考慮されていない立場がある |
| A6 | リスク・コストの片面提示 | リターンのみ提示しリスクを省略、またはその逆 |
| A7 | 次アクションの欠落 | 読者が「で、何をすればいいのか」に到達できない |

**B. 論理（Logic）**

| ID | 観点 | 検出する欠陥 |
|---|---|---|
| B1 | 主張と根拠の飛躍 | 根拠から結論への推論に無断のジャンプがある |
| B2 | 因果と相関の混同 | 相関データを因果として断定している |
| B3 | 出典なき断定 | 数値・事実・仕様を出典なしに断定している（正誤の検証はC1で行う。ここでは出典の欠落を指摘する） |
| B4 | 自己矛盾 | 文書内の別箇所と衝突する記述がある |
| B5 | 定義の曖昧さ | キー概念が未定義または多義的なまま結論に使われている |
| B6 | 過度な一般化 | 限定的な事例・データから全称的な結論を導いている |
| B7 | 循環論法・同語反復 | 結論を前提に使っている、または言い換えを根拠と偽装している |
| B8 | 程度の誇張 | 「大幅に」「劇的に」等、定量的裏付けのない強度表現で結論を支えている |
| B9 | 断定強度の横断不整合 | ドラフトが自ら「未取得」「未確認」「推定」と自認した対象に対し、別箇所で確定的な記述をしている |
| B10 | 宣言と履行の不整合 | ドラフトが冒頭・前提・見出しで宣言した対象範囲・件数・除外条件・構成が、本文で履行されていない |

**B10の実行手順（B4「自己矛盾」では検出できない。B4は"記述vs記述"の衝突を見るが、B10は"宣言vs不作為"を見る）**
1. ドラフトの冒頭・前提・スコープ宣言・見出し・目次から、**約束された内容**をすべて列挙する（対象業界／対象社数／掲載件数／除外条件／章立て）
2. 各約束に対し、本文が実際にそれを履行しているかを照合する。「宣言したのにセクションが存在しない」「n件と書いたが実際はm件」「除外条件を掲げたが適用の痕跡がない」を検出する
3. 検索不要。読者が最初に読む部分と本体が食い違う欠陥は、成果物の信頼性を即座に破壊するため**P0になりやすい**

**B9の実行手順（個別箇所の点検では検出できないため、専用の走査を行う）**
1. ドラフト全体から「未確認」「未取得」「推定」「〜と思われる」「実機未検証」等の自認表現を**すべて列挙**する
2. 自認された対象（機能・画面・データ・範囲）ごとに、本文・表・優先度欄・結論部を**総当たりで**走査し、その対象に対する断定的記述がないか照合する
3. 個別の課題単位ではなく、**成果物全体の断定強度の整合**として見る。検索不要。B9で検出される欠陥は成果物の信頼性の根幹を撃つため、P0になりやすい

**C. 外部照合（Cross-reference）— ドラフトの外側を見る**

A系・B系はドラフトという閉じたテキストの内部整合性しか見ない。C系は**ドラフトの外に出て照合する**。この次元を欠くと、内部的には完璧に整合しているが現実と食い違う成果物を合格させてしまう。

| ID | 観点 | 検出する欠陥 | 手段 |
|---|---|---|---|
| C1 | 外部規格・仕様の引用検証 | 引用した外部規格・法令・ツール仕様・製品仕様（WCAG達成基準とレベル、GA4/GTMの挙動、CMS仕様、検索エンジン挙動等）に誤りがある | **web_search必須** |
| C2 | 参照対象の一次確認 | ドラフトが特定の外部対象（サイト・ページ・原典・既存資料）を論じているのに、批評者がその対象を確認していない。ドラフトの記述と現物が食い違っている／現物にある重要要素をドラフトが見落としている | **web_fetch / 現物確認必須** |
| C3 | 一次ソースの取りこぼし | ドラフトが「該当情報なし」「n件のみ」と宣言している場合、その宣言が実際に成立するか。探索されていない一次ソース・定期刊行物・未採用の候補が存在しないか | **web_search必須** |
| C4 | 成果物標準構成との照合 | ドラフトの成果物タイプ（提案書・ブリーフ・要件定義・監査報告・分析レポート等）を特定し、そのカテゴリなら当然含まれるべき標準構成要素（工程表・マイルストーン・成功指標・スコープ定義等）が欠落していないか | 実務標準知識。検索不要 |

**C3の実行手順（「探索が足りているか」を印象で判断しない）**

C3の失敗は、探索能力の不足ではなく**探索対象の列挙を怠ること**で起きる。以下を機械的に踏む：

1. **対象母集団を先に確定する。** 元要件・仕様・スキル定義に「必須対象リスト」（対象企業n社、対象産業n分野、対象期間等）があれば、それを**全数列挙**する。ドラフトが本文で扱った対象ではなく、**仕様上扱うべき対象の全体**が母集団である
2. **母集団の各要素に対して、ドラフトの掲載有無に関わらず個別にweb_searchを回す。** ドラフトが「A社について該当なし」と書いていても、A社の公式ニュース一覧を実際に見る。掲載済みの対象も、より重要な未採用ソースがないか確認する
3. 母集団が仕様上定義されていない場合は、批評者が母集団の定義（探索すべき一次ソースの範囲）を先に立て、その定義を判定サマリに明示する
4. 探索の結果、取りこぼしがゼロだった場合も**探索した対象名の一覧を証跡として提示する。** 証跡のない「取りこぼしなし」判定は、探索をサボった判定と区別がつかない

**C系の実行規則**
- **モード1（独立批評）ではC1〜C4を必ず実行する。** C1〜C3は検索・実査を伴うため、実行にはツール利用が前提。ツールが使えない環境で起動した場合は、パス1の冒頭で「本批評は対象未確認。C系（外部照合）は未実行であり、実査・検索を要する指摘は網羅していない」と**明示的に宣言**し、ユーザーに実査環境での再実行を要求する。宣言なしにC系をスキップすることは禁止
- C4は検索不要のため、モード2（インライン自己点検）でも実行する
- C1で検出した誤りは、**その誤りが成果物の権威性・判断を歪めるか**でパス2の選別にかける。表現を弱めれば済む程度の誤りはP2以下に落とす

### パス2：選別（偽陽性を落とす）

パス1の候補を1件ずつ以下の3問にかけ、いずれかがNOなら破棄する：

1. この指摘は、想定読者にとって**意思決定または成果物の信頼性に影響する**か？（文体の好みや趣味の問題なら破棄）
2. この指摘は、**ドラフト内の具体的な箇所を指せる**か？（漠然とした印象論なら破棄）
3. この指摘を放置した場合の**具体的な失敗シナリオを一文で言える**か？（言えなければ破棄）

生き残った指摘のみに重大度を付与する。

## 重大度定義

| 重大度 | 定義 | 判定への影響 |
|---|---|---|
| P0 | 結論の妥当性を毀損する。放置すれば成果物として成立しない | 1件でも存在すれば「差し戻し」 |
| P1 | 結論は保てるが、読者からの信頼性・説得力を明確に損なう | 存在すれば「条件付き合格」 |
| P2 | 品質を改善するが、放置しても致命傷にならない | 判定に影響しない。参考情報 |

## 出力フォーマット

以下の構造を厳守する。この順序は「判定→詳細→次アクション」の意思決定フローに対応している。

```

[合格 / 条件付き合格 / 差し戻し] — P0: n件 / P1: n件 / P2: n件
（要件未提供の場合はここでその旨と評価確度への影響を明記）
判定理由を2〜3文で。

## 指摘リスト
| # | 分類 | 対象箇所 | 指摘内容 | 失敗シナリオ | 重大度 | 修正方針 |
|---|---|---|---|---|---|---|

- 対象箇所：セクション名・段落・引用（20字以内）で特定可能にする
- 指摘内容：欠陥の記述。改善提案ではなく欠陥そのものを書く
- 失敗シナリオ：放置した場合に何が起きるか、一文
- 修正方針：方向性のみ。リライト文は書かない
```

### 差し戻しプロンプトの書式

生成側チャットへのハンドオフを摩擦ゼロにするため、以下の形式でコードブロックに入れて出力する：

```
以下は本ドラフトに対する独立批評の結果である。各指摘に対し、(a)修正する、(b)理由を明示して棄却する、のいずれかで応答し、修正を反映した改訂版を出力せよ。指摘の黙殺は禁止。

【P0】
1. [対象箇所] 指摘内容 → 修正方針
【P1】
2. ...
```

P2はデフォルトで差し戻しプロンプトに含めない（差し戻しコストを最小化するため）。ユーザーが「P2も含めて」と指示した場合のみ含める。

## 指摘ゼロの場合

「合格」と判定し、以下を明記する：

- ルーブリックA1〜A7・B1〜B8の各観点を走査した旨
- 特に検証した上で問題なしと判断した観点トップ3（ドラフトの性質上リスクが高かった観点）とその判断根拠

「問題ありません」の一言で終わらせない。走査の証跡がない合格判定は、批評をサボった合格判定と区別がつかないためである。

## 成果物タイプ別の追加観点

ドラフトのタイプを判別し、該当する追加観点をパス1に加える。判別できない場合は汎用ルーブリックのみで実行する。

**UX監査レポート（HCUX等）**
- 指摘と根拠（観察/定量）の対応が1対1で取れているか
- 優先度付けのロジックが一貫しているか（影響度×実装コストの判断基準が指摘間でブレていないか）
- クライアントが反論しうる指摘に先回りの補強があるか

**提案書・企画書**
- 意思決定者が承認をためらう最大の懸念（予算・工数・リスク）に正面から答えているか
- 競合案・現状維持案との比較が明示されているか
- 成功指標と測定方法が定義されているか

**分析レポート（GA4・アクセス解析等）**
- データの計測範囲・期間・除外条件が明記されているか
- 数値の変動に対する解釈が、代替仮説（季節性・計測変更・外部要因）を検討した上でのものか
- 「So What」（数値→示唆→推奨アクション）の連鎖が切れていないか

## 動作モード

本スキルには2つのモードがある。依頼の文脈で判別する。

### モード1：独立批評（デフォルト・フルプロトコル）

上記の批評プロトコル全体（A1〜A7 / B1〜B10 / **C1〜C4**）を実行する。**生成に使ったチャットとは別チャットで起動することが前提。** 納品物・役員向け資料・クライアント提出物は必ずこのモードを通す。

**モード1ではC系（外部照合）が必須である。** C系を省略したモード1は、ドラフト内部の整合性チェックに退化しており、独立批評として不完全である。ツール制約でC系が実行できない場合は、その旨を判定サマリ冒頭に宣言する（黙ってスキップしない）。

#### モード1は2パス構成である（1-S と 1-B）

実証テストにより、**批評の検出力は「批評者が何を知っているか」で決まり、モデルの賢さでは決まらない**ことが判明した。仕様を知る批評者と知らない批評者は、原理的に異なる欠陥を検出する。両方を回収するため、モード1は情報条件を変えた2パスで構成する。**各パスは必ず別チャットで実行する。**

| パス | 渡すもの | 渡さないもの | 検出できる欠陥 |
|---|---|---|---|
| **1-S（仕様参照批評）** | 成果物 + 元要件 + スキル仕様・テンプレート・必須リスト | — | 仕様との照合（必須項目の欠落、ゲート違反、対象リストの取りこぼし）。C3の母集団確定はここでしか行えない |
| **1-B（仕様遮断批評）** | **成果物のみ** | 元要件、スキル仕様、生成経緯、生成に使ったプロンプト、他パスの批評結果 | 白紙の読者の目（B10 宣言と履行の齟齬、素朴な網羅性、成果物の自己完結性、C4 実務標準からの逸脱） |

**1-Bの実行規則**
- 起動時、成果物本体以外の一切の背景情報を要求しない。ユーザーが仕様や要件を提示しようとした場合は**受け取らない**。「本パスは仕様遮断条件で実行する」と宣言して拒否する
- 「元の要件が分からないので評価できない」という留保を出してはならない。**成果物が単独で読者に手渡されたときに成立するか**が1-Bの評価軸であり、要件の不在は欠陥ではなく実験条件である
- A系の評価は「要件に対する充足」ではなく「**この成果物が自ら掲げた目的・宣言に対する充足**」で行う
- 1-Bはノイズ（棄却される指摘）を出しやすい。仕様を知らないため「あるべき像」を自前で補完し、空振りするからである。これは1-Bの欠陥ではなくコストであり、パス2の3問フィルタで削る。**空振りを恐れて指摘を絞ると、1-Bを走らせる意味そのものが消える**

**統合**：1-Sと1-Bの指摘は、ユーザー側で統合する。統合時、**両パスが一致した指摘を「重要」と重み付けしてはならない**（後述の運用ルール参照）。

### モード2：インライン自己点検（生成チャット内・縮約版）

ユーザーの常時指示（メモリ・preferences）または「自己点検して出力して」という依頼により、成果物を生成する同一チャット内で、**出力前に**実行する。

- **適用対象**：成果物ドラフト（レポート・提案書・監査報告・分析・戦略文書・CMS用HTML等）。会話的応答・短い質問回答・コード断片には適用しない
- **縮約ルーブリック**：全項目ではなく、高頻度・高影響の10観点のみ走査する — A1（論点漏れ）/ A4（前提未記載）/ A6（リスク片面提示）/ A7（次アクションの欠落）/ B1（主張と根拠の飛躍）/ B3（出典なき断定）/ B4（自己矛盾）/ **B9（断定強度の横断不整合）** / **B10（宣言と履行の不整合）** / **C4（成果物標準構成との照合）**
- B9・B10・C4は検索不要で実行でき、かつ実証テストでP0級の欠陥が集中した観点であるため、縮約版にも含める
- **手順**：ドラフト完成 → 縮約ルーブリックで自己走査 → パス2の3問フィルタ → P0/P1相当を修正してから出力 → 末尾に1行で証跡を表示：`🔍 自己点検済 — 修正: P0:n / P1:n（修正なしの場合は「指摘なし」）`
- **指摘テーブルは出力しない**（出力の肥大化を防ぐ）。ユーザーが「何を直したか見せて」と求めた場合のみ修正内容を開示する

**モード2の限界（ユーザーに隠さないこと）**：同一文脈での自己点検はself-preference biasを完全には除去できない。モード2はあくまで日常業務の下限品質を引き上げる装置であり、モード1（独立批評）の代替ではない。納品物にモード2だけで済ませることを推奨してはならない。

## 禁止事項

- ドラフトのリライト・改訂版の執筆（差し戻しプロンプト内の修正方針の記述は除く）
- 誤字脱字・表記揺れの指摘（校正スキルの領分。混入すると指摘リストのS/N比が下がる）
- 「全体的に良くできています」等の総評的な賞賛
- 生成側の意図の推測による指摘の自主取り下げ（「意図があるかもしれない」は棄却理由にならない。意図があるなら生成側が差し戻し応答で(b)棄却を選べばよい）
- **納品物の本文に現れない層への指摘**：内部作業ログの記述ミス、スキル仕様書の要件番号の誤記、テンプレート体裁の逸脱など、**成果物を受け取る読者の目に触れない層**の欠陥は出力しない。実証テストでこの層の指摘は全件棄却された。ただし、内部ログが成果物本文に**混入している**場合は、それ自体が読者の目に触れる欠陥なので指摘対象である

## 複数エンジンでクロスチェックする場合の運用ルール

本スキルと外部モデル（ChatGPT等）を併用して二重批評を行う場合、以下に従う。

- **合意項目を「重要」と重み付けしてはならない。** 実証テストでは、両エンジンが揃って指摘した項目の採用率（36%）が、単独指摘の採用率を下回った。複数のLLMが同時に出す指摘は、正しいから一致するのではなく、同じ訓練分布から同じ定型リスク（承認フロー・工数・商標・CMS制約等のテンプレ）を引いているために一致する
- **レビュー順序は「単独指摘 → 合意項目」とする。** 片方だけが出した指摘にこそ、そのエンジン固有の射程が現れる

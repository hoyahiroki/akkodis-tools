# HCUX Audit レポート整形スキル

HCUX Audit（ChatGPTカスタムGPT）または追加プロンプト（改善ロードマップ生成）が出力したMarkdownを、ブルー基調（#0C447C）のA4横向きPDFに整形する。ブラウザ印刷では環境差で崩れる（`@page`による向き固定が用紙回転を誘発する）ため、この環境の Playwright + Chromium で確実に生成する（Path A）。

## 使うタイミング

- ユーザーがHCUX Auditの**監査レポート本体**のMarkdownを貼り、PDF化/整形を求めたとき
- ユーザーが**追加プロンプトの改善ロードマップ出力**のMarkdownを貼り、PDF化/整形を求めたとき
- 「納品用に整えて」「ブルー基調のPDFにして」等、体裁を整える依頼

※このスキルは**整形のみ**。UX評価・課題抽出はGPT側で完了している前提。分析本文（各課題の内容・根拠・改善案）は加筆・要約・改変しない。並び順・体裁のみ整える。

## 手順

1. ユーザーが貼付/添付したHCUX AuditのMarkdownを、そのまま `.md` ファイルに保存する（例：`/home/claude/hcux_input.md`）。**分析内容を編集・要約しない。**
   - 貼るのは**レポート/ロードマップ本文のみ**。「【ご注意】…AIによる自動生成…」等のメタ注記や「作成者向けチェックメモ」行が混じっていたら除外する（納品物に出さない）。
2. レンダラーを実行する：
   ```
   python3 <このスキルのディレクトリ>/render.py <input.md> <output.pdf>
   ```
   - Markdownの先頭に `# タイトル` が無い場合のみ `--title="○○ UX監査レポート"` を付けてタイトル帯を付与する。ある場合は付けない（二重見出しになる）。
3. 生成したPDFを present_files で提示する。
4. 監査レポートとロードマップの**両方**を渡された場合は、それぞれ別ファイルとして2回実行し、2つのPDFを提示する。

## 決定論的な整形ルール（レンダラーが自動適用）

以下は「見た目・並び順の統一」であり、GPTが出力した分析本文は変えない。GPTの出力が実行ごとにブレても、PDF側で一定の体裁に揃える。

- **[A] 改善課題リスト（10列テーブル）の列幅**：`ID` `根拠強度` `影響度` `優先度` を最小幅・折り返し禁止（中央寄せ・バッジ）にし、`問題内容` `根拠` `具体的改善案` `計測指標` を広く取る。列は「見出し名」で役割判定して固定幅を割り当てるため、GPTが列順を変えても崩れない。
- **[B] ロードマップ「含まれる課題」テーブルの列順**：`ID → 優先度 → 影響度 → 根拠強度 → 場所・対象 → 一言要約` に自動で並べ替える。`優先度` `影響度` `根拠強度` は折り返し禁止（中央寄せ・バッジ）。
- **[C] ロードマップ「依存関係・順序の判断」**：入力が**表**で来た場合、`ID○○：…` の箇条書きに変換する（入力が既に箇条書きならそのまま）。表・箇条書きどちらで来ても出力は箇条書きに揃う。
- **[D] 「最短で効果を出す着手順トップ5」**：入力が `ID○○：…` の**箇条書き**で来た場合、`順位／ID／理由` のテーブルに変換する（入力が既に表ならそのまま）。
- **[E] 「評価手法・指標の凡例（参考）」の固定注入**：GPT出力の凡例ブロックを破棄し、**v38.2の凡例と一致する完全版（8ブロック：優先度／影響度／根拠強度／確認レベル／実測根拠未添付時のP0について／ニールセン10原則／認知的ウォークスルー／アクセシビリティ評価の観点）**に差し替える。GPTが実行ごとに確認レベル等を落としても、PDFの凡例は常に完全・同一になる。凡例見出し（「評価手法・指標の凡例（参考）」）を検出したら、以降の凡例本文は次の見出しまで読み捨てる。
- **[F] セクション3「推奨データ収集リスト」のテーブル整形**：GA4／ヒートマップ／追加検証の各テーブル（「関連ID」列を持つ表）を専用スタイルで整え、`関連ID` 列は折り返さない。GPTがセクション3を**3テーブルで出力していれば、そのまま3つの整形済みテーブルになる**。

判定はすべて見出し文字列・列見出し文字列に紐づく決定論的処理。該当しない表・箇条書きは通常どおりレンダリングする。

## レンダラーが“やらない”こと（＝限界。ハルシネーション防止のため）

- **セクション3を1テーブル→3テーブルに分割はしない**：[F]は、GPTが**3カテゴリを別々のテーブル（またはカテゴリ見出し＋表）で出力している**前提で整える。GPTが全カテゴリを1つの表に混在させて出力した場合、各行を「これはGA4／これはヒートマップ」と意味分類して分割するのは推測になるため行わない（1テーブルのまま整形する）。同様に、カテゴリの内容が**表ではなく散文の箇条書き**で来た場合、`データ／何を確認するか／対象ページ／関連ID` の4列を散文から再構成することはしない（列に切り分ける根拠が無いため）。3テーブルを常に保証したい場合は、GPT側でセクション3を3テーブルで出力させる必要がある（現行v38.2は3カテゴリを出力する。楽天レポートの実出力は3テーブルだった）。
- **トップ5の列構成の統一**：[D]は「箇条書き→表」への変換はするが、既にある表の列構成（例：順位/ID/着手内容/理由 か 順位/ID/理由 か）はGPT出力に従う。両実行で列を完全一致させたい場合はGPT側で固定する。

## 保守上の結合（重要）

- **[E]の凡例文言は v38.2 の凡例と一致させてある**（`render.py` の `CANON_LEGEND_BLOCKS`。8ブロックのラベル・順序・本文がv38.2 Instructions凡例と完全一致することを機械照合で確認済み）。GPT側（v38.x）で凡例の文言・ブロック構成を改訂したら、**`render.py` 側の凡例定数も必ず同じ内容に更新する**こと。整形側が凡例の唯一のソースになるため、ここがズレると納品物の凡例が古くなる。
- 固定凡例は**影響度ブロックを含む8ブロック**（優先度／影響度／根拠強度／確認レベル／実測根拠未添付時のP0について／ニールセン10原則／認知的ウォークスルー／アクセシビリティ評価の観点）。GPT出力が旧テーブル形式でも凡例ブロック欠落でも、`render.py` が常にこの8ブロックを固定注入する。したがって `validate.py` のS6（凡例8ブロック未検出）は**入力md側の検出であり、render経由のPDF出力には影響しない**（S6は生md納品時のみ実害。render通す運用なら残置可）。

## 匿名化の扱い（重要）

- **実案件の納品**：匿名化**不要**。レポートは客自身のサイトが対象なので、社名・URLはあって当然。そのまま整形する。
- **公開サンプルを作る場合のみ**：対象サイト名→「大手ECサイト」等の一般語、URL除去、ブランド固有機能名（例：スーパーDEAL、RaCoupon等）→一般語（キャンペーン、クーポン等）、特定できる具体数値→規模表現（例：14,560,602件→数百万件規模）に置換してから整形する。判定基準は「読んだ人が対象を特定できないこと」。

## レンダラー（render.py）の仕様

- 入力：HCUX AuditのMarkdown（見出し `#`〜`####`、パイプ表 `| … |`、箇条書き `- `、太字 `**…**`、URL）
- 出力：A4横向き・ブルー基調・Noto Sans CJK JP埋め込みのPDF
- 自動処理：色付きバッジ（P0/P1/P2・高/中/低・強/弱）、上記[A]〜[F]の列幅・列順・表⇔箇条書き変換・凡例固定注入・セクション3テーブル整形、URL自動リンク化（`/`・`.`等の区切りで折り返す）。
- 依存：Playwright + Chromium（この実行環境に存在）、Noto Sans CJK JPフォント。
- 向きはコード側で横向き指定（`page.pdf(landscape=True)`）。`@page`で向きを固定しない。

## 制約・既知の注意

- 非常に長いレポートはページ数が増えるが、テーブル行は `page-break-inside:avoid` で行の途中改ページを防ぐ。
- GPTの出力が標準的なMarkdown表であることが前提。HTMLテーブル等の特殊記法はパースされない。標準Markdown表に直すようユーザーに促す。
- 実行後、生成PDFの体裁（列見切れ・文字化けの有無）を確認してから提示する。

# ハルシネーション防止ルール

このスキルでは、経営会議で使用されるレポートという性質上、**ハルシネーションは絶対に許容されない**。以下のルールを厳守すること。

## 基本原則

1. **記憶に頼らない**: 必ず `web_search` または `web_fetch` で最新情報を取得してから記述する。Claudeの学習データに基づく情報は使用禁止。
2. **参照元で確認できた事実のみ記述**: 数値・社名・サービス名・日付・固有名詞は、参照元ページで実際に確認できたもののみ記載する。
3. **推測・補完の禁止**: 「おそらく」「〜と思われる」といった表現で情報を埋めてはいけない。不明な点は「情報なし」「記載なし」と明記する。
4. **件数水増しの禁止／早期離脱の禁止**: 指定件数に満たない場合でも捏造しない（水増し厳禁）。同時に、各パートの `references/` に定めた**最低検索回数・全クエリ角度・情報源指定クエリを試し切る前に「件数不足」を宣言することも禁止**。両者は別の違反であり、どちらも犯さない——探索は尽くし、それでも足りなければ正直に件数を報告する。

## 日付検証（ハードゲート）

これは任意チェックではなく、**記事を使用する前の強制通過ゲート**である。
以下のフローを1記事ごとに必ず実行すること。採用可否の基準は `source_policy.md` の「対象月の情報」の定義（(A)一次情報基準／(B)出来事基準）に従う。

### ゲート手順（1記事ごとに実行）
1. `web_fetch` で記事本文を取得する
2. 本文から次の日付を探す:
   - 一次情報の場合: 「発表日」「公開日」「掲載日」「リリース日」
   - 二次報道の場合: 記事の「公開日」に加えて、報じられている「出来事の発生日・発表日」
3. 次のいずれかが対象月の範囲内（例: 2026年5月1日〜5月31日）であることを確認する:
   - (A) 一次情報の発表日／公開日が対象月内
   - (B) 出来事の発生日・発表日が対象月内、**かつ**記事の公開日も対象月内
4. 確認できた場合のみ、そのURLをトピックに使用する
5. 以下のいずれかに該当する場合は**即座に破棄し、別の記事を探す**：
   - 上記(A)(B)のいずれも満たさない（対象月の日付が確認できない）
   - 「更新日」のみが対象月で、一次の発表日・出来事の発生日が対象月外（中身が古い）
   - 前月以前の事象を対象月の記事がまとめ直しただけのもの

### 禁止行為
- 日付確認をせずにURLを採用すること
- 検索結果スニペットの日付だけで判断すること（必ず `web_fetch` で本文確認）
- 「おそらく対象月だろう」という推測での採用

### 検証コストの最適化（誠実性は不変）
候補を `web_fetch` して日付を確認する原則は維持する。ただし空振りfetchを減らすため:
- 一次情報で発行日が結果ページまたは本文冒頭に明確に表示されているもの（公式プレスリリースの発行日、PR TIMESの配信日等）は、その1回のfetchで日付を確定してよい。
- 日付が曖昧なもの・二次情報のみのものは、従来どおり本文を精査し(A)(B)を確認する。
- いずれの場合も「スニペットの日付だけで採用」は引き続き禁止。

## URL検証

- 参照元URLは、実際にアクセスして存在を確認したもののみ記載する。
- 「それらしいURL」を推測で作ってはいけない。
- URLがリンク切れ、または内容と一致しない場合は別の参照元を探す。

## 数値の取り扱い

- 市場規模、成長率、求人倍率、年収、従業員数などの数値は、**参照元に明記されているもののみ**使用する。
- 単位(円/ドル/%/人/件など)は参照元の記載通りに書く。
- 複数の情報源で数値が異なる場合、どちらかを採用する根拠を明記するか、両方併記する。

## 固有名詞の取り扱い

- 企業名・サービス名・プロジェクト名・人名は、**参照元に明記されている正式表記**を使用する。
- 略称・通称を使う場合は、初出時に正式名称を併記する。
- 存在しないサービス名を生成することは絶対にしない。

## 自己チェック

出力前に以下を確認:

- [ ] 全てのトピックに参照元URLが付いているか
- [ ] 全ての参照元が対象月にリリースされたものか
- [ ] 数値・固有名詞は参照元で確認できたものか
- [ ] 不明な情報を推測で埋めていないか
- [ ] 指定件数に満たない場合、正直に報告しているか

1つでもNGがあれば、該当部分を修正または削除してから出力する。

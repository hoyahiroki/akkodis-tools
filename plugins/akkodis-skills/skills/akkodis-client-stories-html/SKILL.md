# AKKODiS client-stories HTML 変換スキル

Word原稿（お客様事例記事）を、AKKODiS `client-stories` 記事ページの `<div class="resource-content">` 配下構造に変換し、**HTML断片を .txt で出力する**。構造は実際の公開 client-stories ページから確定した正規パターンに準拠する。要約・リライト・解説・Markdown・JSONは出さない。

**基本構成が「正」**：全要素がそろった標準的な事例記事の構成を基準テンプレートとし、実際のWord・レイアウトに応じて**要素が無ければ省く／異なれば合わせる**。

**記事カテゴリは3種（アカデミー／ソリューション／コンサルティング）で、カテゴリは必須入力**。カテゴリは主に**上下CTAのDLボタン（右の白ボタン `mod-btn-secondary`）**を決める：アカデミー・コンサルティング＝「事例集をダウンロード」（`akkodis-casestudies`）／ソリューション＝「会社案内をダウンロード」（`corporate-profile`）。左の黄色ボタン `mod-btn-conversion` は全カテゴリ「法人向けご相談」で固定。**カテゴリがプロンプトに無ければ生成前に質問して確認する**（本文構造そのもの＝Q&A/ナラティブ・Index/H3の有無は、カテゴリと別にWordの実態へ追従する）。

**`akkodis-insights-html` とは別物**。client-stories は画像（`mod-lyt-media`系）・Index（`mod-panel-anchor`）・話者ラベル・画像パス（全小文字）が insights とすべて異なる。insightsの構造（`mod-media`/`media_w400`/`mod-box`/末尾プロフィール）を持ち込まない。

## いつ呼ぶか

- AKKODiS `client-stories`（お客様事例）記事原稿（.docx）を、Sitecore投入用HTMLにしたい依頼
- 「client-stories の resource-content に合わせて」「事例記事のHTMLにして」「お客様事例をHTML化」などの指示
- 取材インタビュー形式・三人称ナラティブ形式いずれの事例Word→CMS用HTMLにする場面

## いつ呼ばないか

- HTMLではなく要約・議事録・原稿リライトが欲しい場合
- **対談・インタビュー記事（`trends/insights` 配下）** → `akkodis-insights-html`
- DOCX/PPTX/XLSX等の成果物作成（→ `akkodis-brand-core`）／テキスト校正のみ（→ `akkodis-proofreader`）
- client-stories 以外のページ構造（このスキルは当該構造に依存）

## 入出力の前提

| 項目 | 内容 |
|---|---|
| 入力（必須） | ① Word原稿（.docx, `/mnt/user-data/uploads/`） ② **ページURL**（画像パスの `<slug>`＝URL末尾セグメント。未提供なら生成せず要求） ③ **メタ4項目（原則必須）**：クライアント/担当部署/業界/支援内容。Word本文に無いことが多く別途受け取る。**Wordにも入力にも無ければ生成前に必ず確認**（推測・空欄・捏造で埋めない） ④ **カテゴリ（必須）**：アカデミー／ソリューション／コンサルティングのいずれか。CTAのDLボタンを決める。**プロンプトに無ければ生成前に必ず質問** |
| 実行前確認（必須） | 次のいずれかに該当したら**断片を生成する前に停止して確認する**（このスキルでは推測で進めない）：**(a)** メタ4項目がWordにも入力にも無い → 4項目の値を尋ねる。**(b)** 記事に導入ボックスが無いように見える（三人称ナラティブで導入ボックス・メタが無い記事。**稀なイレギュラー**）→「導入ボックス無しで正しいか」を確認する。**(c)** カテゴリ（アカデミー／ソリューション／コンサルティング）が未指定 → どのカテゴリか尋ねる。確認が取れるまで生成しない |
| 出力（主） | `resource-content` 断片HTMLの **.txt**（`/mnt/user-data/outputs/<slug>-resource-content.txt`） |
| 出力（副） | 記事タイトル（別フィールド貼付用）・構成の判定（本文形式・Index/H3の有無）・アンカー対応表・画像差し替え一覧・確認ポイントを**チャットで**提示（断片には入れない） |
| 断片の契約 | 1行目が厳密に `<div class="resource-content">`、末尾が対応する `</div>`。前後に空行・コメント・Markdown・コードフェンス・BOM・引用符なし。**整形式HTML**（`<p>` 開閉対応。実ソースのバグは再現しない） |

**基本構成・ソースに応じた増減・構成順・各ブロックの正規HTML・話者/段落/キャプション規則・編集判断項目・剥がすもの・セルフチェックは、すべて `references/html-structure.md` にある。変換前に必ず読むこと。**

## ワークフロー

### Step 1: 入力を確認し、原稿テキストを読む

**まずページURLがあるか確認する。** 画像パスの `<slug>` は**ページURLの末尾セグメントをそのまま使う**（例: `…/client-stories/《末尾セグメント》` → `《末尾セグメント》`）。URLが無ければ slug を確定できないので、生成せずユーザーに要求する。

`docx` スキルの手順で本文・順序を把握する。本文順序の保持が最優先。

```bash
extract-text /mnt/user-data/uploads/<file>.docx
```

ここで拾うもの: 記事タイトル（先頭）、リード段落群（＝プロジェクト概要）、Index元（`抱えていた課題：`等の太字見出し＋箇条書き。無い記事もある）、セクション見出し（`■`等の飾り付き）とサブ見出し、話者ラベル（`姓さま：`／`AKKODiS 姓：`。無い記事もある）、地の文と「」引用、画像位置マーカー（`《キャプション》`・`【キャプション】`）、外部リンク（稀）。変更履歴があっても `extract-text` の既定出力＝確定版を使う。

**構成を判定する**（`references/html-structure.md` §2）。基本構成を「正」とし、Wordの実態に合わせて増減する：本文が話者ラベル付きQ&Aか三人称ナラティブか、Index元ブロック・サブ見出しがあるか、を原稿から見極める。判定結果を後でユーザーに提示する。

**メタ4項目（クライアント/担当部署/業界/支援内容）の入力を確認**（原則必須）。Word本文には無いことが多い（表もテキストも無い原稿が実在）。無ければ次の Step 1.5 の確認ゲートで停止して尋ねる。

**カテゴリ（アカデミー／ソリューション／コンサルティング）の指定を確認**（必須。CTAのDLボタンを決める）。プロンプトに無ければ Step 1.5 で停止して尋ねる。

### Step 1.5: 実行前確認ゲート（必須・このスキル固有）

**断片を生成する前に、次のいずれかに該当したら停止してユーザーに確認する。** client-stories は**原則すべての記事に導入ボックス（ロゴ＋メタ4項目＋プロジェクト概要）がある前提**で扱う。ユーザーがWordとページURLだけを貼ってきても、下記が未確定なら勝手に進めない。

1. **メタ4項目（クライアント/担当部署/業界/支援内容）がWordにも入力にも無い** → 4項目の値を尋ねる（原則必須）。空欄・推測・捏造で導入ボックスを埋めない。
2. **記事に導入ボックスが無いように見える**（三人称ナラティブで導入ボックス・メタ情報が無い記事。**稀なイレギュラー**）→「この記事は導入ボックス無しで正しいか（メタ4項目・ロゴ・プロジェクト概要を出さないか）」を確認する。
3. **カテゴリ（アカデミー／ソリューション／コンサルティング）が未指定** → どのカテゴリか尋ねる（CTAのDLボタン＝事例集／会社案内を決めるため必須）。推測で決めない。

確認が取れるまで断片を生成しない。確認後に Step 2 以降へ進む。※通常の userPreferences「野心的解釈で確認せず進める」よりも、**本スキルではこの実行前確認を優先する**（ユーザーの明示指示）。

### Step 2: 画像の位置・キャプション・altを「レイアウト資料」で決める

**画像の位置はWordの `《キャプション》`／`【キャプション】` マーカーと埋め込み画像（`word/media/imageN`）の出現順**から、本文の該当位置に挿入する。マーカー表記は揺れる（`《》`・`【】`等）ので両方拾う。**キャプション有無はマーカーの有無**に従う（マーカー有り＝キャプション付き、後から追加された無マーカー画像＝キャプション無し）。

配置・人物特定の根拠を得るため、必要に応じてWordをレンダリング／埋め込み画像を目視する:

```bash
python /mnt/skills/public/docx/scripts/office/soffice.py --headless --convert-to pdf /mnt/user-data/uploads/<file>.docx --outdir /home/claude/
pdftoppm -png -r 90 /home/claude/<file>.pdf /home/claude/page   # page-01.png … を view で開く
python /mnt/skills/public/docx/scripts/office/unpack.py /mnt/user-data/uploads/<file>.docx /home/claude/unpacked/  # word/media/imageN を alt・人物特定に使う
```

`references/html-structure.md` §8 のとおり、MV画像で各人の顔・服装を把握してから各写真を照合し、alt・キャプションの人物を特定する。**本文中の画像は基本すべて `mod-lyt-media--single`**（左右フロートは提供ソースに無いので作らない。フロートが要るレイアウトはユーザー指示を仰ぐ）。

### Step 3: HTML断片を組み立てる

`references/html-structure.md` の正規スニペットで、基本構成の順（§4）に従って組み、ソースに応じて増減する（§2）。要点:

- **画像パスは全小文字** `/-/media/project/akkodis/akkodis/ja/images/client-stories/<slug>/img-<役割>.webp?la=ja`（insightsの大文字 `Project/Akkodis` とは違う）。`&hash=` は付けない。**srcsetはロゴのみ**。
- **MV**＝`mod-lyt-media--mv`、**本文画像**＝`mod-lyt-media--single`（`mod-lyt-media__img`＞`<img>`、キャプションは `<p class="mod-lyt-media__caption">` で 会社＜br＞部署＜br＞氏名）。
- **導入ボックス** `mod-box-introduction`（`__head`＝ロゴ`__media`＋メタ4項目を `mod-box-introduction__data` ×2、`__body`＝プロジェクト概要の `__data`）→ CTA(top) → **Indexパネル** `mod-panel-anchor`（原稿に見出し＋箇条書きがある場合のみ。中核3セクション、`__item`＞`<a>`内側に`__list`）→ 本文（`<h2 id="anc-NN">`＋`<h3>`＋話者段落／またはナラティブ段落）→ CTA(bottom)。
- **すべての本文H2に連番 `id="anc-NN"`**（Index非掲載セクションにも）。**H3は結合**（Wordで割れていれば1つに）、idなし、サブ見出しが無ければ省く。見出しの飾り（`■`等）は剥がす。
- **話者ラベル（Q&A記事）**: 顧客＝`<strong>姓さま：</strong> `／AKKODiS＝`<strong>AKKODiS 姓：</strong> `（必ず `<strong>`、`AKKODiS《姓》`は`AKKODiS 《姓》`に正規化）。質問＝`<strong>—質問</strong>`。同一話者の続きはラベル無し `<p>`。**三人称ナラティブ記事**は話者ラベルを付けず、実ページ慣習の `<p>`＋`<br /><br />` で「」引用も地の文として入れる（`<blockquote>`不使用）。
- **CTA**: 定型文言・hrefで上下2箇所。1つ目（左）の `mod-btn-conversion`「法人向けご相談」は全カテゴリ共通、ご相談ボタンのみ `target="_blank"`。**2つ目（右）の `mod-btn-secondary` DLボタンはカテゴリで決まる**：アカデミー・コンサルティング＝事例集（`akkodis-casestudies`）／ソリューション＝会社案内（`corporate-profile`）。カテゴリ未指定なら Step 1.5 で確認済みのはず（推測で決めない）。
- **キャプション氏名の強調は実ソースで2通り**（基本 `<strong>` / `<span class="mod-lyt-media__pointText">`）。基本は `<strong>`、実ページが `pointText` なら合わせる。顧客氏名に「さま」、AKKODiS氏名に無し。**説明文型キャプション**（集合写真等）は1行のみ。
- **外部リンク（稀）**は `<a rel="noopener noreferrer" href="絶対URL" target="_blank">`。**IOWN®** はインライン（`<sup>`不使用）。
- `data-__outbound-listener-attached` は出さない。インラインstyle禁止。参照に無いクラスは作らない（例外: キャプションの `mod-lyt-media__pointText`）。**`<p>` は必ず開閉対応**（実ソースのバグを再現しない）。

### Step 4: .txt として保存する

`/mnt/user-data/outputs/<slug>-resource-content.txt`（slugはURL由来）。内容は断片HTMLのみ。

### Step 5: セルフチェックして引き渡す

`references/html-structure.md` §10 を内部実行（非表示）、不一致は保存前に補正。その後チャットで提示:

1. `present_files` で .txt
2. **記事タイトル**（Sitecoreの別フィールド貼付用。断片には入れていない）
3. **構成の判定**（カテゴリ＝アカデミー／ソリューション／コンサルティング、本文形式＝Q&A／ナラティブ、Index・H3の有無、判定理由）
4. **アンカー対応表**（H2 ↔ `anc-NN`。Indexに載せた3つと、載せていないH2を明示）
5. **確認ポイント**: 各画像の**位置の根拠（Wordマーカー＋出現順）と写っている人物**、キャプション本文（会社略称・部署・肩書）、氏名強調タグ（実ページ依存・推定）、alt、ファイル名（`img-…`＝**全て要差し替え**）。slugはURL由来・メタ4項目は入力・カテゴリは入力・配置はレイアウト資料採用、それ以外は推定である旨を明記
6. **除外/特殊処理の申告**: 記事タイトルを断片外にした／メタ4項目を入力から補った／CTAのDLボタンをカテゴリで選んだ（事例集 or 会社案内）／Index・H3を原稿に応じ省いた／H3を結合した／話者ラベルを正規化した、など。Word原稿と公開ページが食い違う可能性（話者・写真の差）があれば、**渡されたWordに忠実に変換した**旨を明記

## 設計判断（実ソース検証＋ユーザー指示で確定／すべて上書き可）

ユーザーが明示指示で覆した場合は指示を優先する。

1. **記事カテゴリは3種（アカデミー／ソリューション／コンサルティング）で必須入力**。カテゴリは主に**CTAのDLボタン**を決める（アカデミー・コンサルティング＝事例集／ソリューション＝会社案内）。未指定なら Step 1.5 で質問。**本文構造そのもの**（Q&A/ナラティブ・Index/H3・導入ボックスの有無）はカテゴリと別に、全要素がそろった標準構成を「正」としてWordの実態に合わせて増減する（本文がナラティブなら話者ラベル無し、Index元が無ければIndex省略、サブ見出しが無ければH3省略、等）。カテゴリと構成判定を提示。
2. **記事スラッグ（画像パス）はページURLの末尾セグメント**（推測しない。URL未提供なら停止）。**画像パスは全小文字**。`&hash=`なし。**srcsetはロゴのみ**。
3. **メタ4項目（クライアント/担当部署/業界/支援内容）は原則必須**。Word本文に無いのが通常→入力で受け取る。**Wordにも入力にも無ければ Step 1.5 で停止して確認**（空欄・推測・捏造で埋めない）。プロジェクト概要はWord冒頭リード段落群。
4. **本文画像は `mod-lyt-media--single`**（中央・標準）。左右フロート用クラスは提供ソースに無いので作らない。
5. **キャプションは2形態**（Wordマーカーの中身で判断）: (a)人物プロフィール型＝会社＜br＞部署・肩書＜br＞氏名（氏名強調は基本 `<strong>`／実ページが `pointText` なら合わせる、顧客に「さま」・AKKODiSは無し）、(b)説明文型＝1行のみ（集合写真・シーン写真等、マーカー文言をそのまま）。**キャプション有無はWordマーカー準拠**。
6. **画像位置はWordマーカー（`《》`/`【】`）＋埋め込み画像順**。本文順に挿入。配置・人物特定の根拠はレイアウト資料（Word・PDF・画像・ユーザー指示）。
7. **すべての本文H2に連番 `anc-NN` id**。Indexパネルに載せるのは中核3セクションのみ。`href="#anc-NN"` と `<h2 id="anc-NN">` を一致。
8. **話者ラベル（Q&A記事）**: 顧客＝`姓さま：`／AKKODiS＝`AKKODiS 姓：`（必ず `<strong>`）。質問＝`<strong>—…</strong>`。insightsの `氏名［以下、略称］：` 方式は使わない。ナラティブ記事は話者ラベル無し。
9. **段落マークアップ**: 基本は段落ごと別 `<p>`。実ページが `<br /><br />` 連結（ナラティブ記事に多い）ならそれに合わせる（「」も内包、`<blockquote>`不使用）。
10. **CTAは resource-content の中**（上下2箇所、定型文言・href）。1つ目（左）`mod-btn-conversion`「法人向けご相談」は全カテゴリ共通で `target="_blank"`。2つ目（右）`mod-btn-secondary` DLボタンは**カテゴリ駆動**（アカデミー・コンサルティング＝事例集 `akkodis-casestudies`／ソリューション＝会社案内 `corporate-profile`）。
11. **末尾の一括プロフィールは無い**（写真直下キャプションが代替）。
12. **記事タイトル・公開日は断片の外**（Sitecore別フィールド）。タイトルはチャットで別提示。
13. **IOWN®はインライン**（`<sup>`不使用）。`™`・複合語IOWNは原稿どおり。**外部リンク（稀）は `rel="noopener noreferrer" … target="_blank"`**。
14. **変更履歴は確定版**（`extract-text` 既定出力）。`--track-changes=all` は使わない。
15. **Word原稿と公開ページが食い違う場合、渡されたWordに忠実に変換する**（既存公開ページに合わせにいかない）。差分は編集判断として確認に回す。

## 禁止事項

- 要約・リライト・解説・Markdown・JSON・コードフェンスを**ファイルに**出すこと
- 参照に無いクラスの創作（例外: キャプションの `mod-lyt-media__pointText`。それ以外の新規クラスは不可）
- insightsのクラス（`mod-media`/`mod-media_left`/`media_w400`/`mod-box`/`img-size-adjustment`/末尾プロフィール）を client-stories に流用すること
- `data-__outbound-listener-attached` 等の実行時JS属性の出力
- インラインstyle、Word由来の色span・装飾span乱立
- 画像パスを `<img>` の外に裸で出すこと（必ず `src` 値としてのみ）
- 画像パスの大文字化（client-storiesは全小文字 `project/akkodis/akkodis`）、`&hash=` の付与、拡張子前の不要な `-`、ロゴ以外へのsrcset
- 実ソースのHTMLバグ（`<p>` 閉じ忘れ・入れ子崩れ）を再現すること（**整形式で出す**）
- Sitecore ID形式リンク、原稿に無い見出し・本文・注釈の補完
- **記事スラッグを推測で決めること**（URL末尾セグメントを使う。URL未提供なら停止して要求）
- **メタ4項目が無いまま導入ボックスを空・捏造で埋めること**（Wordにも入力にも無ければ Step 1.5 で停止して確認）
- **導入ボックスの有無が不確かな記事（稀なイレギュラーの疑い）を、確認せず勝手に生成すること**（Step 1.5 で確認）
- **カテゴリ（アカデミー／ソリューション／コンサルティング）が未指定のまま、CTAのDLボタンを推測で決めて生成すること**（Step 1.5 で質問）
- **`IOWN®` を `<sup>` で囲むこと**（インラインのまま）

変換不能・要確認時（事例構造でない、構成の判定不能、ページURL未提供、メタ4項目未提供、カテゴリ未指定、導入ボックス無しの疑い〈稀なイレギュラー〉等）は、ファイルを作らずチャットで理由を述べて確認・要求する（Step 1.5）。

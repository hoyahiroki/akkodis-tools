# careers/interviews resource-content 構造リファレンス（正本 / 実ページ逆生成・サンプル表記）

AKKODiS careers/interviews 配下の**イベントレポート記事**の `<div class="resource-content">` に出力するHTMLの正本。実際に公開されている Sitecore 出力（`internal-hackathon-2025` / `international-womens-day-2026`）から逆生成した正規パターン。構成順・クラス名・入れ子・空白の扱いはこのファイルを最優先で踏襲する。ここに無いクラスは作らない。

**サンプル中の `《…》` はプレースホルダ**（実原稿の値に置換する）。class 名・タグはそのまま使う構造の一部。

**この構造は `akkodis-insights-html`（trends/insights の対談記事）とは別物。** 主な違い: Index/`mod-box` 無し・アンカー無し・話者ラベル無しの本文が主・H2〜H4を使う・テーブル/箇条書き/受賞列挙/Q&A がある・floated画像は `media_w400`（中央/MVは `img-size-adjustment`）・画像パスは `careers/interviews`。

## 1. 出力コントラクト

- ファイル1行目は厳密に `<div class="resource-content">`、末尾は対応する `</div>`。前後に空行・説明・コメント・Markdown・コードフェンス・BOM・引用符を置かない。
- 画像は必ず完全な `<img …>` 要素。`src` だけを裸テキストで出さない。`<img>` は自己終了 `/>` で閉じる（実ページ準拠）。
- `data-__outbound-listener-attached="true"` を**絶対に出力しない**（ブラウザの追跡JSが実行時に付与する属性）。
- 実ページのHTMLには**オーサリングのバグ**（`</div>` の閉じ過ぎ、孤立した `mod-media-content` ラッパー等）が混在するが、**それらは再現しない**。本リファレンスのクリーンな入れ子で出力する。

---

## 2. 全体の入れ子（出力対象は内側のみ）

公開ページの本文は `resource-content` に入る。**生成・出力するのは `<div class="resource-content">…</div>` だけ**。タイトル・日付・リード文・ハッシュタグ・採用タイルは外（別フィールド or 共通テンプレ）なので断片に含めない（§4）。

```
<section class="resource-body-content">
  … タイトル / 日付 / リード文 / ハッシュタグ（＝出力対象外）…
  <div class="resource-content"> … </div>   ← ★これだけを .txt に出力
</section>
```

---

## 3. 構成順

resource-content 直下に、原稿に存在するものだけを上から順に。イベントレポートは**柔軟な並び**で、固定テンプレではない。

1. **MV画像**（記事先頭の1枚・中央・パターンA）
2. **本文セクション**（H2ごと。配下に H3/H4・本文・テーブル・箇条書き・画像・受賞列挙・Q&A 等が原稿順で並ぶ）
3. **末尾注釈**（`※…`。1つの `<p>`、`<br />` 区切り。複数可）

本文・見出し・画像・表の順序は**原稿どおり**に保持する。画像はセクション先頭に限らず本文の途中にも複数現れる。

> **公開ページは「構造」の正本であって、本文一致の対象ではない。** Word下書きと公開HTMLは編集者の手で文言・見出し・画像配置が改変されている。このスキルの役割は **Word原稿の内容を、本リファレンスの構造へ忠実に変換する**こと。公開ページの編集差分を逆算して再現する必要はない。

---

## 4. 出力しないもの（除外）【MUST】

実ページの `resource-content` 内/周辺に存在しても、**以下は出力しない**:

- **ハッシュタグ**（`#Fusion Activators #生成AI …` のような行）。
- **「Fusion Activators」とは 等のPromo/関連サービス誘導ブロック**（見出し＋本文＋リンク）。
- **採用タイル**（新卒採用／キャリア採用／障がい者採用。`careers/common/img-tiles-*` を使う共通テンプレ）。

これらは Sitecore の別フィールド or 共通コンポーネントであり、記事断片の責務外。

## 5. ブロック別の正規HTML

画像パスは `/-/media/Project/Akkodis/akkodis/ja/images/careers/interviews/《slug》/img-《役割》.webp?la=ja`。**`《slug》` はページURLの末尾セグメントと同一**（§7・§9）。`&hash=…` は公開時に Sitecore が付与するので生成不要。**拡張子直前に `-` を付けない【MUST】**（§7）。

### 5-0. ルート ＋ A. MV画像

```html
<div class="resource-content">
<p><img class="img-size-adjustment" alt="《記事タイトル》" src="/-/media/Project/Akkodis/akkodis/ja/images/careers/interviews/《slug》/img-mv.webp?la=ja" /></p>
```

- MV の `alt` は**記事タイトル**。
- ハッシュタグ行は**出力しない**（§4）。MV を resource-content の先頭に置く。

### 5-1. 見出し（H2 / H3 / H4）

```html
<h2>《見出し》</h2>
<h3>《小見出し》</h3>
<h4>《孫見出し》</h4>
```

- **プレーンな `<h2>`〜`<h4>`。`id` も class も付けない**（careers はアンカー目次を持たない）。
- 見出しレベルは原稿のマーカーから判定する（§6）。

### 5-2. 本文段落・スペーサー

```html
<p>《本文》</p>
<p>《1段落内の改行は》<br />
《br で繋ぐ》</p>
<p>&nbsp;</p>
```

- 1意味段落＝1 `<p>`。段落内の改行は `<br />`。
- セクション境界やブロック間の間隔として `<p>&nbsp;</p>` を実ページ準拠で挟む（多用しない。原稿/レイアウトに従う）。

### 5-3. イベント概要（2バリアント・原稿の書式に従う）

**バリアント1: テーブル**（原稿が表形式のとき）

```html
<table class="mod-table mod-table-1row_narrow">
<colgroup>
<col class="mod-wide-30per" />
<col />
</colgroup>
<tbody>
<tr>
<th>《項目》</th>
<td>《値》</td>
</tr>
<tr>
<th>《項目》</th>
<td>《値（複数行は》<br />
《br で改行）》</td>
</tr>
</tbody>
</table>
```

- class は `mod-table mod-table-1row_narrow` 固定。`<colgroup>` は1列目 `mod-wide-30per`・2列目クラス無し。`<thead>` は使わず `<tbody>` のみ。左列＝`<th>`（項目）、右列＝`<td>`（値）。

**バリアント2: ■リストを1つの `<p>` に**（原稿が箇条の行書きのとき）

```html
<p>
■《項目》：《値》<br />
■《項目》：《値》<br />
■《項目》：《値》
</p>
```

- 原稿が `■項目：値` の行で概要を書いている場合はテーブル化せず、1つの `<p>` に `<br />` 区切りで流す。マーカー（■）は原稿どおり保持。

### 5-4. 箇条書き（`<ul>`）

```html
<ul>
<li>
《見出し的ラベル》<br />
《説明本文》
</li>
<li>
《見出し的ラベル》<br />
《説明本文》
</li>
</ul>
```

- **プレーンな `<ul>`/`<li>`。class は付けない**（Index 用の `mod-list-point__item` とは別系統＝careers 本文の箇条書きは無印）。
- 「ラベル＋説明」型の項目は、ラベル行のあと `<br />` で説明を続け、1つの `<li>` に収める。

### 5-5. 登壇者紹介（`<p>` リスト）

```html
<h3>登壇者紹介</h3>
<p>《イニシャル等》：《経歴本文》<br />
《続きの経歴》</p>
<p>《イニシャル等》：《経歴本文》<br />
《続きの経歴》</p>
```

- 登壇者1名＝1 `<p>`。先頭の `イニシャル：`（全角コロン）はそのまま本文に含める（`<strong>` で囲まない＝実ページ準拠）。複数行は `<br />`。

### 5-6. 審査員ブロック（強調ラベル＋導入＋名簿）

```html
<p><strong>審査員</strong><br />
《導入文（例: 審査を担当したのは以下の6名の皆さんです。）》</p>
<p>《氏名》（《所属・肩書》）<br />
《氏名》（《所属・肩書》）<br />
《氏名》氏（《社外社名・肩書》）<br />
《氏名》氏（《社外社名・肩書》）</p>
```

- 見出し的な強調ラベル＋導入を1つの `<p>` に（ラベルは `<strong>`、`<br />` で導入文）。
- 名簿は別の `<p>` に `<br />` 区切り。**社外の人物は氏名に「氏」を付ける／AKKODiS は付けない**（§8）。

### 5-7. 受賞・チーム列挙（賞ごと）

```html
<h3>《【賞名】チームN「チーム名」等》</h3>
<p><img class="img-size-adjustment" alt="《alt》" src="/-/media/Project/Akkodis/akkodis/ja/images/careers/interviews/《slug》/img-《役割》.webp?la=ja" /></p>
<p><strong>《プロダクト名：「…」等の強調行》</strong></p>
<p>《説明本文》</p>
<p>＜《審査員名》講評＞<br />
《講評本文》</p>
```

- 賞の見出しは `<h3>`。直下に中央画像（パターンC・本文無し＝`<p><img>`）。
- 「プロダクト名：「…」」のような強調行は `<p><strong>…</strong></p>`。
- 講評は「＜○○講評＞」をラベルとして1つの `<p>` に、`<br />` で講評本文を続ける。

### 5-8. インタビュー Q&A

**複数話者のセクション**（例: 川口・成田）— 各回答に話者ラベルを付ける:

```html
<p><strong>―《質問》</strong></p>
<p><strong>《姓》：</strong>《回答》<br />
《続き》</p>
<p><strong>《姓》：</strong>《回答》</p>
```

**単独インタビュイーのセクション**（導入文で1名を名指し。例: 奥田）— 回答は**話者ラベル無し**:

```html
<p>《導入：…に、お話を伺いました。》</p>
<p><img class="img-size-adjustment" alt="《alt》" src="…/《slug》/img-《役割》.webp?la=ja" /></p>
<p><strong>―《質問》</strong></p>
<p>《回答（ラベル無し）》<br />
《続き》</p>
<p><strong>―《質問》</strong></p>
<p>《回答（ラベル無し）》</p>
```

- **質問は1つの `<p>` 全体を `<strong>`** で囲み、先頭に `―`（ダッシュ1つ）を置く。**原稿が `――`（2倍ダッシュ）でも `―` 1つに正規化する【MUST】**（公開ページに合わせる）。
- 回答: **そのセクションに話者が複数いれば** `<strong>《姓》：</strong>` を付ける。**導入で1名だけ名指しされている単独セクションでは付けない**（プレーンな `<p>`）。

### 5-9. イベント写真（中央・既定）— パターンC（本文無し）

```html
<p><img class="img-size-adjustment" alt="《alt》" src="/-/media/Project/Akkodis/akkodis/ja/images/careers/interviews/《slug》/img-《役割》.webp?la=ja" /></p>
```

- **careers の画像は中央/全幅が既定。** イベント写真・人物写真とも、原則この形（`<p><img class="img-size-adjustment">`）で本文順の出現位置に置く。
- `alt` は文脈（直近の見出し・キャプション・写っている人物）から付ける。原稿が装飾画像として扱っている場合は `alt=""`（実ページに空altの例あり）。alt は要確認（§9）。

### 5-10. floated 画像ブロック（左右配置・レイアウト資料で判定）— パターンB

横に寄せて配置された写真（コメントブロック等）。**配置はレイアウト資料で決める（§7）。**

右配置の例（左配置は `mod-media_left`、構造同一）:

```html
<div class="mod-media mod-media_right">
<div class="mod-media-media">
<img class="media_w400" alt="《alt》" src="/-/media/Project/Akkodis/akkodis/ja/images/careers/interviews/《slug》/img-《役割》.webp?la=ja" />
</div>
<div class="mod-media-content">
<p class="mod-txt-em">《サブ見出し》</p>
<p>《本文》<br />
《続き》</p>
<p class="mod-txt-em">《次のサブ見出し》</p>
<p>《本文》</p>
</div>
</div>
```

- ラッパーは `mod-media mod-media_left|_right`。中の `mod-media-media` に `<img>`。
- **floated 画像のクラスは `media_w400`【MUST】**（中央/MVは `img-size-adjustment`）。
- **本文・サブ見出しは `mod-media-content` で包む【MUST】**。コメント群の小見出しは `<p class="mod-txt-em">`。
- 同種の小見出しでも、画像の横に入らず単独で段落間に立つ場合は `<h4>` になることがある（実ページに両例あり＝編集判断・§9）。

### 5-11. 末尾注釈（`※` ・複数対応）

```html
<p>※１　《注釈1本文》<br />
※２　《注釈2本文》<br />
※インタビュー内容、所属は取材当時のものです。</p>
```

- **1つの `<p>` に全注釈、`<br />` 区切り**（`mod-footnotes` div は使わない）。
- **`※インタビュー内容、所属は取材当時のものです。` は定型クロージングとして必ず最後に入れる【MUST】**（原稿に記載が無くても出す）。
- 原稿に番号付き注釈（`※１` 等）が他にあれば、それらを上に積み、最後に上の定型行を置く（すべて同じ `<p>`・`<br />` 区切り）。注釈が無ければ定型行のみの1行 `<p>`。
- マーカー（`※`／`※１`／全角スペース等）は**原稿の表記を保持**。

### 5-12. 外部リンク

```html
<a rel="noopener noreferrer" href="《絶対URL》" target="_blank">《表示テキスト》</a>
```

- 本文中・注釈中いずれも同形式。URLは絶対URL（Word の `word/_rels/document.xml.rels` の `TargetMode="External"` から取得）。`data-` 属性は付けない。

### 5-13. ルート閉じ

末尾でルートを閉じる `</div>`。

## 6. 見出しレベルの読み取り（Word原稿の規則）

careers の入稿Wordは**見出しレベルを手書きマーカーで指定**している。代表的な2方式があり、どちらも「H1/H2/H3/H4」というレベル指定を含む。**マーカーと足場を剥がし、レベルでタグを決め、クリーンな見出しテキストを中身にする。**

- **方式A（接頭辞グルー型）**: 見出しテキストに直付け。例 `## **H2開催の背景**` → `<h2>開催の背景</h2>`（`##`・`**`・`H2` を除去）。
- **方式B（欄ラベル型）**: レベル指定が独立行のフィールドラベルで、**次の行**が実テキスト。例:
  ```
  **H2(以下、入力ください)**
  **開催背景**
  ```
  → `<h2>開催背景</h2>`（ラベル行と `(以下、入力ください)` 等の足場・`**` を除去し、次行を見出しに）。

補足:
- **H1 はタイトル**。`resource-content` には入れず、タイトルフィールド（断片の外）扱い（§9 でチャット提示）。
- **「本文(以下、入力ください)」直後の段落はリード文**。これも断片の外（リードフィールド）。
- **すべての見出しにマーカーがあるとは限らない**。`アイデアソンの流れ` や `【優勝】…` のように、原稿では**ただの太字行**のことがある。その場合は**入れ子の文脈からレベルを推定**（H2 セクション配下の小見出し＝H3、その下＝H4）。推定したレベルは §9 で要確認として提示。
- 「メタディスクリプション」「OG画像」「公開URL」「タグカテゴリ」「公開日」等の入稿フィールドは**本文ではない**。断片に含めない（必要に応じてタイトル/リードと併せてチャットで参考提示）。

## 7. 画像の規則（パス・ファイル名・配置）

- **パス**: `/-/media/Project/Akkodis/akkodis/ja/images/careers/interviews/《slug》/img-《役割》.webp?la=ja`。
  - **パスセグメントは `careers/interviews`**（insights の `trends/insights` ではない）。
  - **`《slug》` はページURLの末尾セグメント**（推測しない。URL未提供なら停止して要求）。
  - 大文字小文字は `Project/Akkodis/akkodis`（実ページ準拠）。`?la=ja` を付け、`&hash=…` は付けない。
- **ファイル名の `-` 規則【MUST】**: **拡張子の直前に `-` を付けない**。`img-mv.webp`（○） / `img-mv-.webp`（×） / `img-kochi-hitomi-2026.webp`（○） / `img-kochi-hitomi-2026-.webp`（×）。実公開ページには末尾dash付きが散見されるが、本スキルは**末尾dash無しを必須**とする。
- **ファイル名は仮置き（要差し替え）**: 実アセット名は記事ごとに大きく揺れる（年号付き・slug前置き・大文字混在等）。役割から `img-《役割》.webp` を意味のある名前で仮置きし、§9 で**要差し替え**と明示。
- **配置（中央/左右）はレイアウト資料で決める【MUST】**（§9）。**既定は中央/全幅**（`<p><img class="img-size-adjustment">`）。**横寄せ portrait はレイアウト資料（Wordのレンダリング／PDF／レイアウト画像／ユーザーの明示指示）が横配置を示す場合のみ** floated（§5-10）にする。XMLの `align`・写っている人数では決めない。ユーザー指示があれば最優先。
- **画像クラス: 中央/MV＝`img-size-adjustment`、floated（左右）＝`media_w400`【MUST】**。
- `alt` は写っている人物・直近見出し・キャプションから付ける（単独人物＝「会社 人名」、社外ゲストは「氏」付き／AKKODiS は無し）。装飾扱いなら空 alt。altは要確認。

## 8. 話者ラベル・リンク・全角の規則

- **インタビュー質問**: 1つの `<p>` 全体を `<strong>` で囲み、先頭に `―`（ダッシュ1つ）。**原稿が `――` でも `―` 1つに正規化【MUST】**。
- **回答の話者ラベル**: そのセクションに話者が**複数**いる場合のみ `<strong>《姓》：</strong>`（全角コロン）を付ける。**単独インタビュイー（導入で1名名指し）のセクションでは付けない**。
- **「氏」**: 社外（非AKKODiS）の人物は、名簿・写真alt・初出で氏名に「氏」を付ける。AKKODiS の人物は付けない。
- **太字/斜体**: `<strong>` / `<em>` のみ。Word由来の色span・装飾span・サイズspanは捨てる。`mod-txt-em` は floated コメントブロックの小見出し専用（§5-10）。
- **全角・記号**: 原稿の全角コロン `：`・全角スペース・`■`・`＜＞`・`※` 等は原稿どおり保持。Word由来のゼロ幅スペースや不要な全角二重スペースはノイズなので落としてよい（§9 で軽微補正として申告）。
- **`&nbsp;` スペーサー**: ブロック間に `<p>&nbsp;</p>` を実ページ準拠で配置（原稿/レイアウトに従い、多用しない）。
- **IOWN®**（出現時）: `IOWNⓇ`/`IOWN®` は ® をインラインのまま `IOWN®`（`<sup>` で囲まない）。`™` もインライン。複合語IOWN（`IOWN構想` 等）は原稿の付け方を保持。判断に迷う商標は `akkodis-proofreader`/`akkodis-brand-core` の固有名詞ルールに従い、原稿の文言を保持する。

## 9. 入力で決まる項目／編集判断になる項目

**初回プロンプトには Word原稿（.docx）と「ページURL」の両方が必要**。画像パスの `《slug》` は**ページURLの末尾セグメントをそのまま使う**。URLが無ければ停止して要求する。

| 項目 | 取得元 | 確認 |
|---|---|---|
| 記事スラッグ（画像パス） | **ページURLの末尾セグメント**（推測しない） | URL必須。未提供なら停止 |
| 見出しレベル | 原稿のマーカー（§6）。マーカー無しは入れ子から推定 | マーカー無しは要確認 |
| 画像の配置（中央/左右） | **§7: レイアウト資料（Word・PDF・画像・ユーザー指示）。既定は中央。横寄せは資料が示す時のみ floated（MUST）** | 原則そのまま採用 |
| 画像の alt | 写っている人物・直近見出し・キャプション。装飾は空alt | 要確認 |
| 画像ファイル名 | 役割から `img-《役割》.webp`（**末尾dash無しMUST**）を仮置き | 要差し替え |
| イベント概要の形式 | 原稿が表＝テーブル / `■`行＝`<p>`（§5-3） | 原稿に従う |
| 注釈 | 原稿の `※` 行（複数可）＋定型クロージングを必ず付与（§5-11・MUST） | 本数のみ要確認 |
| 話者ラベルの要否 | セクションの話者が複数＝付ける / 単独＝付けない（§8） | 文脈判断 |

**スラッグと配置は推測の編集判断ではない**（スラッグ＝URL、配置＝レイアウト資料）。alt・ファイル名・見出しレベル（マーカー無し時）・注釈本数は最善推定＋確認提示。

## 10. 出力前セルフチェック（内部実行・非表示）

保存前に確認し、不一致は自動補正してから保存する（結果は表示しない）。

1. 1行目 `<div class="resource-content">`、末尾は対応する `</div>` か
2. 画像が `<img>` の外に裸で出ていないか。`<img …/>` が自己終了か
3. **画像クラスが 中央/MV＝`img-size-adjustment`／floated＝`media_w400` になっているか**
4. floated が `mod-media mod-media_left|_right` ＞ `mod-media-media`(`media_w400`)＋`mod-media-content`、サブ見出しは `mod-txt-em`、になっているか（閉じ過ぎ div・孤立ラッパーが無いか）
5. 見出しが `<h2>`〜`<h4>` のプレーン（**id・class 無し**）か。原稿のマーカー/足場（`H2`・`(以下、入力ください)`・`**`）を剥がしたか
6. テーブルが `mod-table mod-table-1row_narrow` ＋ `colgroup`(`mod-wide-30per`/無印) ＋ `tbody` ＋ `th`/`td` か
7. 箇条書きが**無印の `<ul>`/`<li>`** か
8. Q&A: 質問が `<p><strong>―…</strong></p>`、回答ラベルが複数話者のみ `<strong>姓：</strong>`・単独は無し になっているか
9. 注釈が**単一 `<p>` + `<br />`**（複数注釈も同一 `<p>`・`mod-footnotes` div 無し）で、**末尾に定型クロージング `※インタビュー内容、所属は取材当時のものです。` が必ず入っているか【MUST】**
10. **除外物（ハッシュタグ・Promo「…とは」・採用タイル）を出していないか**【MUST】
11. 外部リンクが `<a rel="noopener noreferrer" href="…" target="_blank">` 形式で `data-` 無しか
12. 画像srcの `《slug》` がページURL末尾と一致しているか。パスセグメントが `careers/interviews` か。**拡張子直前に `-` が無いか**【MUST】
13. `data-__outbound-listener-attached` を出していないか
14. インラインstyle／参照に無いクラス／Markdown／コードフェンス／説明文が無いか
15. resource-content 内にタイトル・日付・リード文を入れていないか

ルート要素や `<img>` を正しく生成できない場合、またはページURLが無い場合のみ、ファイルを作らずチャットで理由を述べて確認・要求する。
